Why I did that? 


1. I created isdelete on the userModel for soft delete so that we can see the deleted users for auditing.


2. JWT Role Decision -> In the JWT I am storing _id and role and when the user makes a request the middleware will query the db to check the role and is active status for confirmation. Yes it would add latency in production but when we store the role in JWT and we only query the Db for isActive status that would create a problem when the Admin downgrades/upgrades the role of that particular user. Because the JWT expire in 24hr. So the user may have access when downgraded or he does not have access when upgraded so I made this decision. 


3. Assignment Implementation
 Query Every time -> Currently I am querying every time when the user makes an API call. The queries are checking if the User isActive, getting his role, etc... it would add latency but for the simplicity i am currently doing this.

3. Production scenario 
In a production scenario we can store the permissions and user info like {email ,role, isActive} in redis so that when the user makes a request we can check the users info without querying DB and if the Admin makes changes the permissions or makes changes in user data like change in role or isActive status before updating we can check if the user data is in redis cache if yes delete the cache so that next time the user visits the role of the user will be queried from the DB which is updated and the user can access or loose access as per the update or become inActive. It also applies for Role and Permissions 


4. I did not use Express-Mongo-Sanitize because the mongodb has built in NoSQL injection prevention at Schema level instead i used strict type checking like (typeof name === 'String') for extra security.


5. I used  await bcrypt.hash() instead of bcrypt.hashSync()  because while hashSync will block the entire thread process the await will hands the process to a worker thread and stops that particular function to stop for that process to complete while other request are handled as usual  


6. Assignment Implementation 
 In this Assignment for simplicity I Hardcoded the RBAC roles. I used enum in the {role} field in userModel and when the user wants to access a resource/endpoint the middleware(jwtAuthorize.js) first checks the JwtToken and then in the router I hardcoded the parameter for the (authrize.js) middleware in each router that is required authorisation to access the endpoint.

6. Production scenario:
 But then I thought what if the user changes the permission then i would have to update the entire DB. So for production grade, secure and better scalability we split the role into a seperate model and we only store the _id of the role in the User Model and when the user wants to access a endpoint that needs permission we check the user role and get the data from the Role model and join it with the user data and check the permission and user active status and is the user is soft deleted then we check if the user role has the specific permission for this endpoint. If yes the request moves on to the respective controller. If not the middleware throws a 403 Unauthorised to the user. Simple. 


7. I used the timestamp option in userModel and roleModel to keep log of the created and lastupdated date and time.
